    • Generating Test Data

      • All-Purpose Test Data: One of the easiest to use realistic data generators publicly available on the Internet is mockaroo which also has a very generous free tier. It can generate, at the time of writing this, 143 type of data ranging from names, addresses and NHS numbers to avatar URLs, drug names and company names. Mockaroo also supports regular expressions, so users can create their own data types.

        In order to generate data, simply add the required columns, select an output type and then either download or preview the results. The output type is a matter of personal preference; I personally lean towards JSON, however a table format such as Excel may be easier to look at for most people.

        The number of results is defined by the numbers of rows set to generate. Only up to 1000 rows can be generated as part of the free tier, however that is more than enough for most non-enterprise users.

        Another very powerful tool for generating test data is faker.js, which has the advantage of being open source. Setting it up involves a bit of technical complexity, however comprehensive instructions on setup and usage can be found on the faker.js GitHub page. The library is also available as an NPM package.

      • Bank Cards: Bank card details that can be used for testing purposes are provided by the companies that develop the products meddbase integrates with. Online payments in meddbase currently go through SagePay which provide a wide range of test card details usable for test transactions.

      • Phone Numbers: Mobile numbers in the range 07700 900000 to 07700 900999 can be used for testing purposes as these are reserved by Ofcom, which is the regulator for communications services in the United Kingdom. Ofcom also provides a range of geographic landline numbers that can be used for testing.

        Please note that none of these phone numbers are allocated to communications providers, so calling or messaging them will not work, however they are in the correct format and using them ensures that calls or text messages are not received by real people which is very important because text messages or automated calls may potentially contain sensitive test data.

      • Email Addresses: There are many websites that provide disposable email address services. Some of the most notable ones are tempail.com, tempmail.net and temp-mail.org. These can be used securely to test email sending functionality to and from meddbase.

    • Logging Defects

      • Structure

        The structure of a defect card in targetprocess should be in accordance with the following guidelines:

        The body of the defect starts with a detailed description. Please be sure to include all the relevant information, while avoiding extra unneeded information that may confuse the reader.

        The description is separated from the steps to reproduce by a horizontal line. This can either be inserted via the user interface of the editor, or by adding a <hr> tag while inspecting the source of the defect body.

        The second section contains details on where the issue reproduces and what the steps to reproduce are. Credentials and environment variables are typed in on a single line in the format “system > user > password (optional) > browser (optional)”. The browser is only required if the issue is browser specific. Examples include “system.meddbase.com > MMS.User@HCA > Mozilla Firefox” or “qa.meddbase.com > BRL.Vlad > myPassword”. Following this line are the steps to reproduce, which must be typed on sequentially numbered rows. Please don’t skip any logical step, while also making sure that the STR are short and concise, as not to confuse the reader. Ideally, the steps to reproduce should be a list element; HTML lists can be inserted into the defect description via the text editor user interface.

        If additional information is available, such as a stack trace, then it is separated from the previous sections by another horizontal line per block of additional information. Most often, stack traces will preserve the formatting of the HTML element that they were copied from; to avoid copying this formatting to the defect body, please first remove it. One of the easiest ways of accomplishing this while also preserving newline characters is to paste into a text document, select all, copy again, and finally paste into targetprocess.

      • Attachments

        It is preferable that attachments are captured via Greenshot (for image) and OBS Studio (for video); this is explained in more detail in the Tools section of this handbook. Video attachments can only be uploaded as files, while image attachments can be either uploaded as files or pasted in comments as inline images. The golden rule is: If an inline image takes up too much screen space, then upload it as a file instead. Too much space would be an image that would cause the comment box to expand in height more than twice the default size.

        When uploading attachments, please make sure that the file name is a timestamp in the format "YYYY-MM-DD_HH-MM-SS". This can be configured in the software product used to capture the attachment.

      • Defect Parameters

        There are many built-in as well as custom fields that can be set while logging a defect. The most commonly used ones are the following:

        • State

          Throughout its lifecycle, a defect will transition through several states.

          • New: A newly logged defect. Not yet confirmed so it could potentially be invalid, a duplicate, it may need more information, etc.
          • Confirmed: A defect that has been confirmed to be valid by one of the developers.
          • In Progress: A developer has started fixing this defect.
          • Awaiting Merge: A developer has finished fixing this defect and is now waiting for the changes to be merged to the branch that will be deployed to the environment on which the defect needs to be retested.
          • Parked: A defect ends up in this state if something causes the development work to be put on hold for an undetermined amount of time. An example of why this may happen could be that a defect was logged for a component that will soon be replaced.
          • Resolved: The defect has been fixed, and the fix has been deployed and is available for retesting and regression testing. Defects previously in the Awaiting Merge state are automatically moved to the Resolved state once the changes have been deployed; this works via an integration between targetprocess and GitLab.
          • Awaiting Merge To Production: This state is for defects that have been fix verified and that are in the queue to be deployed to production sites.
          • Done: This state is automated in the same way that the Resolved state is; once the fix has been verified, while the defect is in the Awaiting Merge To Production state, the card will automatically be marked Done by the GitLab integration after the changes have been deployed to production sites.

          Some states require the user to leave a comment when moving the card to that state. This comment is plain-text only, so if tags, URLs, screenshots etc. need to be included in this comment then please enter the text-only part first so that targetprocess allows changing the state and then go back to your comment and update it with any edits that may be required.

        • Assignments

          Teams can be assigned to defects and, in most cases, it will only be possible to assign a single team per defect. Assigning a team to a defect, and generally to a targetprocess entity, can kick off a team workflow which can divide a state into multiple sub-states, as can be seen below. Only assign teams to targetprocess entities if instructed to do so by the QA manager.

          Apart from team assignments, individual people can be assigned to specific roles. More than one person can be assigned to a single role. Only assign individual people to targetprocess entities if instructed to do so by the QA manager.

          The people assigned to the role that is responsible for the defect’s current state will appear in a heavier font.         • Severities & Business Values

          The severity and business value of a defect dictate the priority of fixing that defect. Defect severities are 'set in stone', while business values are determined by a Risk Assessment Matrix.

          The seven severities currently defined in targetprocess are the following:

          • Enhancement: This severity is reserved for nice-to-have features. Defects that usually fall into this category are quality-of-life changes that enhance the user's experience. Occasionally, missing features that would improve on an MVP (Minimum Viable Product) may fall into this category as well.
          • Trivial: This severity is mostly for typos and minute cosmetic defects.
          • Minor: Reserved for minor functionality issues and more noticeable cosmetic defects.
          • Average: The default defect severity. Average is used for garden-variety functionality issues.
          • Major: Reserved for major functionality issues. These defects will usually throw an exception.
          • Critical: The defects that fall into this category are severe performance issues that cause the application to freeze or hang, runtime failures that cause the application to crash, serious functionality issues that prevent users from progressing with their work, exploitable security holes, data loss, data corruption, defects with associated clinical risk, et cetera.
          • Blocker: This is reserved for defects that prevent the progress of internal testing or for showstoppers that are impeding a rollout.

          Business values are defined by the following Risk Assessment Matrix. Each business value is determined by the combination of the defect’s severity and its likelihood of occurrence.

        • Custom Fields

          Custom fields are defined in the targetprocess admin configuration section, and these can be of several types such as drop-down list, free-text field, boolean control, computed field (calculated from the values of other fields by a defined formula) etc. The most notable custom fields are:

          • Resolution: Defines the reason for moving defects into certain states. Common uses of the Resolution field include:
            • After a developer fixes a defect, they will mark it Resolved and set the Resolution to Fixed.
            • When a developer believes the defect is invalid, they will mark it Resolved and set the Resolution to Invalid.
            • After a defect fails retesting, a QA person will move it back to the Confirmed state and set the Resolution to Reopened.
            • When a developer needs more information on a defect from a QA person, they will mark it Resolved and set the Resolution to Need More Info.
            Several other values are available for the Resolution field, however these are less commonly used.
          • Support Ticket ID: If a request for QA review includes a support ticket ID, then that ID needs to be copied over to the defect logged off the back of that request. More information about this is available in the section detailing the Requests For QA Review workflow.
          • State Of Fix: The value of this field never needs to be set manually; it is controlled by an integration with GitLab, and the value will automatically be set to “Fix is on Development” when the fix for the defect has been merged to the development branch and to “Fix is on Development and Production” when the fix for the defect has been deployed to the production environment.

        • Relations

          Relations in targetprocess are symmetrical links that connect two entities. They can be of one of five types: Relation, Dependency, Blocker, Link and Duplicate.

          A relation between two targetprocess entities is similar to a flow, so if a defect was logged off the back of a request, for instance, then the defect will be linked by an outbound relation from the request.

          Because relations are symmetrical, the above means that in the defect context the request is linked by an inbound relation to the defect.

          An easy way to determine the context of one or more relations is to think of the grey circle at the middle of the diagram as the current entity. In our examples above, in the request context the defect is an outbound entity while in the defect context the request is an inbound entity.

      • Grammar/Punctuation/Syntax/Semantics

        • Grammar: Please always try to use correct grammar when logging defects and/or creating comments. To facilitate this, the Grammarly add-on can be installed in your browser. More information about Grammarly is available in the Tools section of this document. Examples of commonly made grammatical errors include:
          • The incorrect use of homophones such as Then/Than, Their/There/They’re, To/Too, Your/You’re, Of/Off.
          • Using nouns instead of verbs. Examples: a) the user is unable to login the user is unable to log in; b) the user is experiencing issues attempting to setup the configuration the user is experiencing issues attempting to set up the configuration; c) we will rollout a new version of the application we will roll out a new version of the application;
          • Incorrectly capitalized acronyms or initialisms. Examples: a) Json JSON; b) Asap ASAP;
        • Punctuation: Correct punctuation matters, so always try to keep on top of that. Grammarly, most of the time, can help with this as well. Examples of commonly made punctuation mistakes include:
          • Not ending sentences with a period, or an exclamation/question mark where appropriate.
          • Using dashes for hyphens and vice versa. A hyphen (-) is a punctuation mark that’s used to join words or parts of words, whereas a dash (–) is longer than a hyphen and is commonly used to indicate a range or a pause
          • Using apostrophes in plural words. Examples: a) SMS's SMSs; b) ID's IDs;
        • Syntax: Syntax is essentially about what word comes before and after another word. Please make sure your sentences don't end up sounding like tongue twisters. English can be weird; it can be understood through tough thorough thought, though.
        • Semantics: Semantics are concerned with the meaning of signifiers such as words, phrases, signs, and symbols. In terms of meaning, some very important guidelines on logging defects are:
          • Never log any defects that sound like requests for the developer. An incorrect defect title would be “Please change the color of this text.”, however the correct title could be "Inconsistent text color being used.".
          • Never be subjective while describing a defect. An incorrect description would be “When I logged in, I was presented with an error message.”, however the correct description could be “After logging in, it was observed that an error occurs.”. Always try to avoid describing defects in the first person, and always try to be as objective as possible.
          I will end this section with a programming joke: A woman asks her husband, a programmer, “Could you please go shopping for me, and buy one carton of milk? And if they have eggs, please get six.” A short time later, the programmer comes back with six cartons of milk. His wife asks, “Why did you buy six cartons of milk?” to which the programmer replies, “They had eggs.”.

      • Comments

        While creating comments in targetprocess, it is possible to tag users or entities in the comment body.

        Tagging users can be done by inserting an @ sign followed by starting to type the user’s name and then selecting them from an auto-complete list; doing so will cause the tagged user to receive a notification of the newly created comment.

        Tagging entities can be done by typing “ID: ” followed by the targetprocess ID of the entity.

        Tagged users and entities will resolve to interactive elements as soon as the comment has been saved. These can now be clicked to navigate to the tagged user and/or entity.

        While tagging allows users to send notifications to other users by including them in the comment, another way of sending notifications to other users that a new comment has been created is to use the checkboxes under the comment editor. By ticking these, email notifications can be sent to the card owner, the people assigned to the card, and/or the members of the project in which the card was created.

        Lastly, in order to make the best use of the comments section, please make sure that you are using the comment sort order that is right for you. This is a matter of personal preference and only needs to be done once, as it is a per-user setting and not a per-card setting. I personally prefer oldest first, because it works better for me to be able to go through the comment history from the top to the bottom.

    • Continuous Integration/Delivery/Deployment

      • Continuous Integration: Developers practicing continuous integration merge their changes back to the main branch as often as possible. The developer's changes are validated by creating a build and running automated tests against the build. By doing so, you avoid the integration hell that usually happens when people wait for release day to merge their changes into the release branch.

        Continuous integration puts a great emphasis on testing automation to check that the application is not broken whenever new commits are integrated into the main branch.

      • Continuous Delivery: Continuous delivery is an extension of continuous integration to make sure that you can release new changes to your customers quickly in a sustainable way. This means that on top of having automated your testing, you also have automated your release process and you can deploy your application at any point of time by clicking on a button.

        In theory, with continuous delivery, you can decide to release daily, weekly, fortnightly, or whatever suits your business requirements. However, if you truly want to get the benefits of continuous delivery, you should deploy to production as early as possible to make sure that you release small batches that are easy to troubleshoot in case of a problem.

      • Continuous Deployment: Continuous deployment goes one step further than continuous delivery. With this practice, every change that passes all stages of your production pipeline is released to your customers. There's no human intervention, and only a failed test will prevent a new change to be deployed to production.

        Continuous deployment is an excellent way to accelerate the feedback loop with your customers and take pressure off the team as there isn't a Release Day anymore. Developers can focus on building software, and they see their work go live minutes after they've finished working on it.

        • To put it simply continuous integration is part of both continuous delivery and continuous deployment. And continuous deployment is like continuous delivery, except that releases happen automatically.
